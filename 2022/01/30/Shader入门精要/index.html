渲染流水线

什么是渲染流水线？

渲染流水线可以简单的理解为计算机将三维物体转换到二维屏幕渲染的一个过程。
渲染流程分为三个阶段：应用阶段、几何阶段、光栅化阶段。

  • 应用阶段
    这个阶段主要由应用主导，通常由 CPU 负责实现。
    在这一阶段主要由三个任务：

    1. 数据准备：摄像机，场景中的模型，光源
    2. 剔除：将一些不需要渲染的部分剔除出去，提高渲染性能
    3. 设置渲染状态：材质，纹理，shader 等
      输出渲染图元给下一个阶段—–几何阶段

  • 几何阶段
    这一阶段通常在 GPU 中进行，逐顶点、逐多边形的操作。
    几何阶段最主要的任务就是把顶点坐标变换到屏幕空间中，
    再交给光栅器进行处理。
    这一阶段会输出屏幕空间的二维顶点坐标，以及对应的深度值、着色等相关信息。

  • 光栅化阶段
    这一阶段也是在 GPU 中进行的。
    通过几何阶段产生的数据，决定每个渲染图元中的哪些像素应该被绘制在屏幕上。
    通过计算每个图元覆盖哪些像素，并为这些像素进行插值计算去算出它们的着色。

CPU 和 GPU 之间的通信

渲染流水线的起点是 CPU，即应用阶段。应用阶段大致可分为下面三个阶段：
（1）把数据加载到显存中

（2）设置渲染状态

（3）调用 DrawCall

几何阶段：

  • 顶点着色器
    完全可编程，通常用于实现顶点的空间变换、顶点颜色等功能。另外，有一些能在顶点计算的，
    则尽量在顶点中计算好，因为在片元着色器计算的话，相当于每一个片元都得执行一次运算。

  • 曲面细分着色器
    是一个可选的着色器，用于细分图元

  • 几何着色器
    可选着色器，可以被用于逐图元的着色操作，或者用于产生更多的图元。

  • 裁剪
    这个阶段的目的是将那些不在摄像机视野内的顶点裁剪掉，并提出某些三角图元面片。这个阶段是可配置的。

  • 屏幕映射：
    不可编程和不可配置，负责把每个图元的坐标转换到屏幕坐标中。

光栅化阶段：

  • 三角形设置
    GPU 固定实现
  • 三角形遍历
    GPU 固定实现
  • 片元着色器
    完全可编程，这一步中的片元，是已经经过 GPU 对三角形计算覆盖像素，并且对其中的属性进行过插值计算的
  • 逐片元操作
    高可配置的。负责执行很多重要操作，例如修改颜色、深度缓冲、混合等。

顶点着色器

顶点着色器是流水线第一阶段，它的输入来自 CPU。处理的数据是顶点数据，也就是每一个顶点，都会执行一次顶点着色器，顶点着色器本身不能创建或者销毁顶点，也不能获取顶点之间的关系。
顶点着色器一般会用来计算坐标转换、逐顶点光照等，也可以计算一些后续阶段需要用到的数据。

屏幕映射

屏幕映射的任务是把图元的 x 和 y 坐标转换到屏幕坐标系。屏幕坐标系是一个二维的坐标系。屏幕映射不会对 z 做处理，z 坐标系会和屏幕坐标系一起构成一个坐标系叫窗口坐标系，一起被传递到光栅化阶段。

三角形设置

由这一步开始，就进入光栅化阶段，光栅化的第一步就是三角形设置。光栅化需要计算三角形图元对像素的覆盖情况，而三角形设置就是根据上一阶段传过来的顶点信息，去计算三角形边界的像素信息输出给下一个阶段。

三角形遍历

三角形遍历阶段会检查每一个像素是否被一个三角形网格所覆盖。如果被覆盖，就会生成一个片元，三角形遍历阶段，会根据上一个阶段的计算结果，来判断一个三角形网格覆盖了哪些像素，并使用三个顶点的顶点信息对整个覆盖区域的像素进行插值。

片元着色器

片元着色器是另一个非常重要的可编程着色器阶段。
前面的光栅化阶段，实际上不会影响到屏幕上的每一个像素的颜色值，而是会产生一系列的数据信息，用来表示三角形网格是怎样覆盖每个像素的，每个片元就负责存储这样一系列数据。
真正对像素产生影响的是阶段的下一步操作————逐片元操作。

片元着色器的输出，是上一阶段对顶点信息插值得到的结果。

片元仅可以操作单个片元，不能去作用于自己的邻居。每个片元的计算和顶点类似，都是相互独立的，但是片元可以访问导数信息。

因为 GPU 在光栅化的时候，一般是以 2x2 像素为基本单位，这里 GPU 会去计算相邻像素的变化率，即 ddx 和 ddy。

逐片元操作

在 DirectX 中，这一步也被叫做输出合并阶段。
这一步有几个主要任务。
（1）决定每个片元的可见性，例如深度测试、模板测试。
（2）如果一个片元通过了所有测试，就需要将片元的颜色值和已经存储在颜色缓冲区的颜色进行混合。
逐片元操作是高度可配置的，我们可以去设置每一个操作的细节。

一个片元，只有通过了所有的测试之后，才能和颜色缓冲区的颜色进行混合，最后再写入颜色缓冲区。

在模板测试中，一个片元不管有没有通过测试，都可以根据配置去选择要不要写入模板缓冲区的值。
而在深度测试中，只有通过深度测试的片元，才有权利选择要不要写入深度缓冲。并且深度测试通过的标准，也是可配置的，例如可以设定大于算通过也可以设定小于或者等于才算通过。

如果一个片元通过了以上所有测试，那么就可以来到合并操作了，当然，对于不透明的物体渲染，显然是可以直接关闭合并的。

合并的规则也是可配置的

Early-Z

虽然逻辑上，以上的测试都是在片元着色器后面进行的，但是大多数 GPU 都会尽可能在执行片元着色器之前就进行测试。因为在某些情况下，有可能一些片元着色器计算完之后根本不需要被渲染，提前测试的话，就可以省去一些无用的计算，从而提高性能。例如提前进行深度测试，这种提前的深度测试，也被叫做 Early-Z。

但是，如果将测试提前做，其检验结果可能会与片元着色器中的一些操作冲突。例如，如果我们在片元着色器上进行透明度测试，调用 clip 函数来手动舍弃片元，就会导致 GPU 无法提前执行各种测试。
（这里所谓的冲突，可以体现在片元通过提前测试并且写入了缓冲，但是在片元着色器却被 discard 掉了，导致缓冲区的结果已经不可靠了，因此就没办法再相信这个缓冲区去做提前测试了）

因此，现代的 GPU 会判断片元着色器中的操作是否和提前测试发生冲突，如果有冲突，就会禁止提前测试。

双重缓冲

当模型的图元通过层层计算和测试之后，就会显示到屏幕上。GPU 采用双重缓冲策略，对于场景的渲染，是离屏渲染的，即在后置缓冲区，但渲染完毕之后，GPU 则会交换前置缓冲区和后置缓冲区来显示新的内容。

问题

  1. 什么是 OpenGL/DirectX？
    这两者本质上都可以理解为 GPU 的 API

  2. 什么是 HLSL、GLSL、Cg？
    在可编程的着色器阶段，需要用特定的着色器语言去编写逻辑。常见的着色器语言有 DirectX 的 HLSL，OpenGL 的 GLSL，以及 NVIDIA 的 Cg。

  3. 什么是 DrawCall？

    DrawCall 本身的含义就是 CPU 调用图像编程接口去命令 GPU 进行渲染的操作。
而 DrawCall 容易引发性能瓶颈的元凶则是 CPU。

    3.1 CPU 和 GPU 是如何实现并行工作的？

    CPU和GPU是通过命令缓冲来实现并行工作的，其中DrawCall就是其中一种命令，
    其他的命令还有改变渲染状态，切换纹理等。


        3.2 为什么 DrawCall 多了会影响效率？

    CPU在提交DrawCall需要一系列的准备工作，而GPU的渲染能力很强。
    如果DrawCall的数量过多，CPU就会花费大量时间用来准备，造成CPU过载。


        3.3 如何减少 DrawCall？
    方法有很多，但最常见的则是批处理的方法。

  4. 什么是固定管线渲染？

    在一些比较老旧的 GPU 中，流水线仅提供配置操作。
